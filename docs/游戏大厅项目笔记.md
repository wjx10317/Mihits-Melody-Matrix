## 房间与数据结构

### Q1: 单个房间最多加入多少人？

**2人参与对局，旁观不限人数。**

| 角色 | 枚举      | 人数 | 棋子     |
| ---- | --------- | ---- | -------- |
| 房主 | `_host`   | 1    | 黑棋     |
| 玩家 | `_player` | 1    | 白棋     |
| 旁观 | `_spec`   | 不限 | 只看不下 |

`userlist` 是 `std::list<QPair<int,QString>>`，无容量上限。UI 只有两个座位。

### Q2: 为什么用 `std::list` 而不是 `std::vector` 或 `unordered_map`？

| 操作      | `list`          | `vector`      | `unordered_map` |
| --------- | --------------- | ------------- | --------------- |
| 插入/删除 | O(1)            | O(n)          | O(1)            |
| 遍历      | O(n) 缓存不友好 | O(n) 缓存友好 | O(n)            |
| 按ID查找  | O(n)            | O(n)          | O(1)            |

**实际选择理由**：频繁加入/退出房间，`list` 的 O(1) 插入删除合适。但说实话数据量小（几十人），用哪种差别不大。如果要优化，`unordered_map<int, QString>` 按 ID 查找更高效。

### Q3: 服务端用链表存房间用户，尾部旁观者最晚收到落子，有性能问题吗？

**链表串行广播确实存在尾部延迟偏高的问题。** 最坏情况下一个慢连接的 TCP 缓冲区满了会拖慢后面的发送。

**修复方案**：每次广播前拷贝到 `vector` 随机打乱顺序，或记录 `broadcast_offset` 从不同起点轮转。

**但实际影响可忽略**：五子棋落子频率极低（几十秒一着），单次消息体只有几个字节，非阻塞发送下微秒级差异无感知。

## 密码安全

### Q4: MD5 加密真的安全吗？

**不安全，三层硬伤：**

| 问题                       | 后果                                                |
| -------------------------- | --------------------------------------------------- |
| MD5 已被碰撞破解（2004年） | Carnegie Mellon 宣布 MD5 "cryptographically broken" |
| 无盐                       | 相同密码→相同密文，彩虹表秒破                       |
| 单次哈希                   | 现代 GPU 每秒数百亿次 MD5，6位数字密码 <1ms         |

### Q5: 怎么改进？

| 方案                           | 防彩虹表 | 防暴力 | 推荐度 |
| ------------------------------ | -------- | ------ | ------ |
| MD5（现状）                    | ❌        | ❌      | —      |
| MD5 + 固定盐                   | △        | ❌      | —      |
| SHA-256 + 随机盐 + PBKDF2 迭代 | ○        | △      | ○      |
| **bcrypt / scrypt / Argon2**   | ✅        | ✅      | ✅      |

### Q6: 固定盐 vs 随机盐？

- **固定盐**：代码里写死 `SALT="mihits2024"`，攻击者拿到代码后重新跑彩虹表 → 自欺欺人
- **随机盐**：每个用户注册时独立生成随机字符串，不保密，明文存库。作用是把「算一次哈希暴打全库」变成「每个用户单独暴力破解」

### Q7: 盐的工作原理？

```
注册：
  salt = random_bytes(16)
  hash = SHA256(salt + password)
  存库: | salt(明文) | password_hash |

登录：
  查出该用户的 salt
  hash = SHA256(salt + 用户输入的password)
  比对 hash 和库里的 password_hash
```

**bcrypt 更简单**：盐和迭代次数编码在结果字符串里，调用方完全透明。

### 在哪端加盐？盐怎么存储?

在客户端加盐，根据用户注册的账号密码生成一个随机字符串就是盐，然后与计算后的md5摘要一起发送，服务端接收落在数据库中，但是还是有问题，盐还是暴露在中间路径下，升级为SHA256也不行，就是比md5长一倍，本质还是个摘要算法，不能加解密。

### 所有密码相关哈希算法横向对比

#### 1. PBKDF2

- 减速手段：单纯提高迭代次数
- 内存占用：极低
- 抗 GPU 爆破：一般
- 适用：政企、等保合规、老旧兼容系统

#### 2. bcrypt

- 减速手段：加密循环迭代
- 内存占用：低
- 抗 GPU：中等
- 特点：自带盐、存储字符串一体化，开发简单

#### 3. scrypt

- 减速手段：高迭代 + 大量内存读写（内存硬算法）
- 内存占用：很高
- 抗 GPU：很强，显卡显存不够无法批量破解

#### 4. Argon2id（当前最优）

- 减速：迭代 + 内存 + 并行度三维可调
- 内存占用：可控
- 抗 GPU/ASIC：最强，OWASP 推荐新标准

以上都为慢哈希，人为增加运算成本

#### 5.MD5和SHA256

都是快速哈希摘要算法，输出固定长度大小摘要

为什么加盐也不可以?因为现代运算速度很快，gpu一秒钟几十亿次计算，可以暴力破解，必须使用慢哈希

### Q19: 对象树机制你有了解吗？

Qt 所有 `QObject` 派生类自带**对象树（Object Tree）**：

创建对象时传入 `parent` 父对象，形成父子层级；父对象销毁时，**自动递归 delete 所有子对象**，实现内存自动管理，不用手动释放堆对象。

底层实现原理

1. 每个 QObject 内部维护两个变量

   - `QObjectList children`：保存所有直接子对象；
   - `QObject* m_parent`：指向唯一父对象（一个对象只能有一个父）。

2. 构造函数 

   ```
   QObject(QObject *parent = nullptr)
   ```

   

   传入非空 parent 时，自动执行：

   ```
   parent->children.append(this)
   ```

   ，完成挂载。

3. 析构函数流程（关键）

   

   父对象执行 

   ```
   ~QObject()
   ```

   ：

   1. 遍历 `children` 列表；
   2. 逐个调用 `delete child;`；
   3. 子对象析构时，会把自己从父的 children 链表移除；
   4. 递归销毁整棵子树，深度优先。

4. 动态修改父子关系

   

   ```
   setParent(QObject*)
   ```

   ：

   - 先从旧父的 children 移除自己；
   - 挂载到新父的 children；
   - parent=nullptr 脱离对象树，需手动 delete。

常见自动销毁场景比如关闭窗口，父对象手动 delete，栈区父对象超出生命周期

由于父子关系的存在，所以**禁止双向互相设父**，**挂入对象树的子对象，绝对不能手动 delete**，**所有子控件必须堆分配 new，父可栈 / 堆**，栈对象不能作为子节点。主要应用于GUI界面的管理

### Q20: 信号槽机制底层？

信号槽是qt独有的机制，主要依靠预处理工具moc反射实现；首先要使用该机制的类必须声明qobject宏，在预处理期之前解析宏生成moc_xxx.cpp,他会解析信号，槽函数等标记，将信号转化为函数，生成静态对象，里面包含槽，信号，参数类型，索引；一个分发函数qt_metacall,根据索引反射调用槽函数，一个虚函数metaObject，暴露当前类的元对象的指针。

然后首先是连接，分别获取发送者和接受者的元对象，通过通过函数指针查表，信号，校验参数。sender内部会维护一个信号连接链表，链表么个节点存储接受对象的指针，槽索引，连接类型，参数校验标记；

发射信号时会读取自身元对象，找到绑定的对应槽索引，根据连接类型，判断发送信号的方式，分为直连(发射既立刻调用，阻塞然后执行完毕)仅允许收发同线程使用，跨线程使用如果操作ui会直接崩溃，这种情况采用队列连接，信号发送后如果是阻塞队列链接该线程休眠，直至接受线程处理完槽函数，容易死锁，但是是多线程下操作ui的唯一方式，还有自动连接，自动根据情况匹配，接受者在收到调度后根据qtmetacall通过索引找到函数地址，依靠moc的反射逻辑，调用槽函数

任意QObject对象析构时，会自动遍历自身所有信号，槽连接:

1.作为发送者:通知所有接受者删除对应连接；

2.作为接受者:从所有发送者的链表中移除自身绑定；

避免传统回调的悬空/野指针，不会出现对象销毁后调用回调触发内存访问崩溃，解耦，支持多对多，原生跨线程安全，声明周期安全，传参灵活(但是必须可拷贝)

连接类型可以指定，是connect的第五个参数直接连接信号参数值可以传引用，队列连接因为异步，不允许传引用，因为它要求值可拷贝(直接连接就可以用引用避免拷贝)，拷贝过去之后如果变量已经被释放，则会访问野指针造成崩溃，但是右值引用两者都会崩溃，绝对不能用

因为ui强制规定跑在主线程，所以接收槽为ui时一定要队列连接，不能直连。

### Q21:队列连接时怎么依靠事件队列的?事件队列循环机制具体是怎么回事?

触发时机：emit 信号时

当发送者与接收者不在同一线程、连接类型为 `Qt::QueuedConnection`（Auto 跨线程自动切这个）：

1. 执行 `emit 信号(参数)`，进入 moc 生成的信号函数；
2. 不会立刻调用槽函数，而是做两件事：
   - 把信号所有参数**完整拷贝一份**（必须可拷贝，所以不能裸引用）；
   - 构造一个 `QMetaCallEvent` 事件对象，里面保存：接收者指针、槽函数索引、拷贝后的参数列表；
3. 调用 `QCoreApplication::postEvent(receiver, event)`，将事件投递到**接收者对象所属线程的事件队列尾部**；
4. `emit` 执行完毕，非阻塞队列连接可以让发送线程代码继续向下运行，完全不阻塞，发送线程和槽执行解耦。

2. 接收线程何时处理这个调用事件

接收线程自己的 `QEventLoop` 循环不断从队首取事件处理：

1. 循环取出 `QMetaCallEvent`；
2. 调用全局分发器 `QCoreApplication::notify(接收者, 事件)`；
3. notify 识别这是元调用事件，取出槽索引和拷贝好的参数；
4. 调用 `receiver->qt_metacall(索引, 参数数组)`，通过 moc 反射执行槽函数；
5. 槽执行完成，销毁事件对象，事件循环继续取下一条事件。

1.每个线程独立拥有事件队列

每一个创建的 `QThread` 内部都维护独立私有事件队列；主线程 main 函数的 `QApplication` 绑定主线程队列。

- UI 控件全部归属主线程，只能主线程事件循环处理绘制、鼠标、窗口事件；
- 子线程必须调用 `thread.exec()` 开启自身事件循环，否则投递到子线程的事件永远不会被处理。

2. 事件循环 QEventLoop 核心工作逻辑

`app.exec()` / `thread.exec()` 本质是开启阻塞式循环，标准循环步骤：

1. 检查当前线程的事件队列是否为空；

   - 空：线程休眠，释放 CPU，不占用算力；
   - 非空：取出队列最前面的事件；

2. `notify(receiver, event)` 全局分发入口：把事件交给目标对象；

3. 目标对象执行 

   ```
   event(QEvent *e)
   ```

    函数，根据事件类型分流：

   - 鼠标 / 键盘 → keyPressEvent /mouseMoveEvent
   - 窗口重绘 → paintEvent
   - 定时器 → timerEvent
   - 跨线程信号槽 → QMetaCallEvent（执行槽）

4. `event()` 返回 true：事件已消费，不再向上冒泡；返回 false：事件冒泡到父对象继续处理；

5. 事件处理完毕，释放事件内存；

6. 回到循环开头，重复读取队列；

7. 调用 `loop.quit()` 标记循环退出，exec () 返回，循环终止。

3. 事件分类，都会进入对应线程队列

1. 系统窗口事件（键鼠、窗口大小、关闭）：操作系统消息转换为 QEvent，推入主线程队列；
2. 绘制事件 `QPaintEvent`：调用 `update()` 不会立刻绘制，仅往队列插入绘制事件，连续 update 会自动合并减少绘制次数；
3. 定时器事件：QTimer 到期，给目标对象投递定时事件；
4. 跨线程信号槽：前面说的 `QMetaCallEvent`；
5. 自定义事件：`postEvent()` 手动投递自定义 User 事件。

### Q22:Qt中QDialog和QMainWindow的区别是什么？

**QMainWindow**：主窗口框架，用来做程序**顶层主界面**，是整个软件的根窗口。

**QDialog**：弹窗 / 对话框，用于弹出临时交互面板、提示、设置、子窗口，依附主窗口存在。

QMainWindow 固定分层布局（自带完整框架）

内部预设一套标准桌面程序结构，必须通过 `setCentralWidget()` 设置中心业务控件：

1. 菜单栏 menuBar ()
2. 工具栏 addToolBar ()
3. 状态栏 statusBar ()
4. 停靠部件 dockWidget
5. 中心窗口 centralWidget（必须手动填充业务 UI）

适合完整客户端主界面

QDialog 无内置框架，空白容器

没有菜单栏、状态栏、工具栏、停靠栏；

只能直接放置布局、按钮、输入框等控件，轻量化弹窗载体，适合设置面板、登录弹窗、棋谱查看窗口。

QDialog 独有阻塞接口 `exec()`

调用 `dialog->exec()` 会启动**嵌套局部事件循环**，模态阻塞，冻结其他窗口交互，关闭弹窗后函数才返回；

同时支持 `show()` 非模态无阻塞展示。

QMainWindow 无 exec () 方法

只能调用 `show()`，永远是非模态，复用全局主事件循环，不存在阻塞逻辑，不能用来做弹窗。

QDialog 原生支持 `setWindowModality()` 设置模态 / 非模态：

- Qt::ApplicationModal：阻塞整个程序

- Qt::WindowModal：仅阻塞父窗口

  QMainWindow 一般作为父窗口，极少设置模态，不用于弹窗阻塞交互。

  QDialog

  - 配合 `setAttribute(Qt::WA_DeleteOnClose)`，关闭自动释放堆内存；
  - 栈对象临时弹窗也可快速使用，用完自动销毁。

  QMainWindow

  全局唯一顶层主窗口，生命周期贯穿程序全程，一般在栈创建，程序退出时自动销毁，不会频繁新建销毁。

### Q23:什么是模态、非模态交互，项目中为什么通常用非模态？

一、基础定义

1. 模态对话框（Modal）

调用 `dialog->exec()` 弹出，特点：

1. 阻塞当前调用处的事件循环，代码停在 `exec()` 等待对话框关闭才继续往下执行；
2. 阻塞同一应用内其他所有窗口，用户无法点击、操作其他界面；
3. 独占输入焦点，必须关闭弹窗才能操作主窗口。

底层原理：

`exec()` 内部新建局部 `QEventLoop`，开启独立事件循环，接管消息处理；弹窗关闭后调用 `loop.quit()`，函数返回，恢复原事件循环。

2. 非模态对话框（Modeless）

调用 `dialog->show()` 弹出，特点：

1. `show()` 立即返回，不阻塞原有主线程事件循环；
2. 弹窗和主窗口可自由切换，两边同时操作、并行交互；
3. 共用主线程唯一全局 `QEventLoop`，所有窗口事件统一排队处理。

二、两者核心区别

1. 阻塞特性

- 模态：阻塞调用代码、阻塞其他窗口交互；
- 非模态：无任何阻塞，主逻辑持续运行。

1. 事件循环

- 模态：嵌套局部事件循环；
- 非模态：复用主线程主事件循环。

1. 使用入口

- 模态：`exec()`；
- 非模态：`show()`。

1. 适用场景

- 模态：确认弹窗、登录弹窗、强制填写表单（必须处理完才能继续）；
- 非模态：设置面板、日志窗口、对战记录、调试面板（可边看弹窗边操作主界面）。

### Q24:多线程是怎么做的?

由于ui层要跑在qt主线程，接收数据为了开销(非阻塞轮询cpu开销大)需要设置为阻塞，所以我给接收数据用_beginthreadex单独创建了一个线程，但是还是有问题，发送数据和UI主线程还在一起，数据量很大时会导致阻塞，不过目前客户端发送数据的规模还不算很大，目前可以接受，后面如果数据规模大起来了，就会把发送线程单独扔出来，

### Q25：Qt中多线程如何实现？

一般是QThread + moveToThread，写一个工作类(必须定义Q_OBJECT宏)，定义一个工作函数以及可以使用的成员变量，然后定义qthread指针对象，worker对象，通过movetothread方法将对象管理权转移给子线程句柄，然后绑定槽函数，thread->start()启动线程，这个子线程就有自己的事件循环。

### Q26:为什么采用_beginthreadex而不是qt支持的Qthread呢?

因为是给tcp阻塞接收数据创建线程吗，在网络层，这种偏底层的模块需要注重可移植性，因此调用原生api。我学习qt的时候做过im的demo，网络模块就是这样直接移植过来的。

### Q27:网络层为什么不用qtcpsocket？

就是出于解耦考虑嘛，以及qtcpsocket对tcp有额外封装，其异步通信极度依赖事件循环机制，需要考虑可移植性给非qt客户端调用时需要拷贝整个qt依赖，对于服务端来说，qtcpsocket没暴露底层epoll语句无法自定义事件，调度等，而且qtcpsocket跨线程使用有很多麻烦，QTcpSocket 内部依赖 `QSocketNotifier` 监听操作系统 socket 读写事件，通知器**绑定创建时的线程事件循环**；一旦在其他线程调用 `write()`、`connectToHost()`、`read()`，会触发报错。

### Q28:ui是怎么工作的？

`.ui` 是 **XML 格式的界面描述文件**，由 Qt Designer 可视化拖拽生成，存储控件类型、大小、位置、层级、属性、布局、信号槽关联、样式等所有界面信息，不能直接被 C++ 编译器识别，需要工具编译转换。

Qt 提供预处理工具 `uic.exe`，核心作用：读取 xml 格式 `.ui`，自动生成标准 C++ 头文件 `ui_xxx.h`。

编译流程（qmake/cmake 自动执行）：

1. 构建工具扫描项目内所有 `.ui` 文件；
2. 调用 `uic 文件名.ui -o ui_文件名.h`；
3. 生成的 `ui_xxx.h` 内部包含一个命名空间 `Ui` + 界面类（如 `Ui::MainWindow`）。

界面类中持有所有控件指针：`QPushButton *pushButton; QLineEdit *lineEdit;`；

内置 `setupUi(QWidget *widget)` 核心函数：

- 创建所有控件实例；
- 设置尺寸、文字、样式、布局；
- 绑定 Designer 里拖拽关联的信号与槽；

内置 `retranslateUi()` 多语言翻译函数，处理控件显示文本；

控件统一挂载到传入的父窗口上。

优点

1. 可视化拖拽，不用手写大量控件布局代码，开发速度快；
2. 统一管理界面属性、多语言翻译；
3. 布局自动适配窗口拉伸，减少坐标硬编码；
4. 编译期转换为 C++，无运行时解析 xml 开销，性能等同于手写控件。

缺点

1. 修改.ui 后需要重新构建项目，uic 才会更新 ui_xxx.h；
2. 复杂动态界面（运行时动态增减大量控件）不适合完全依赖 ui 文件；
3. 深层自定义控件需要提升控件（提升为自定义类），配置稍繁琐。(五子棋就是这么做的)

这也是我第二个游戏项目没有使用QT的原因

### Q28:协议结构怎么设计的？为什么选择采用这种形式？有什么缺点?

我采用tcp连接加自定义二进制协议的形式进行封包，数据传输。自定义包相对于json、protobuf、xml等方式，更加灵活的使用，且调试时字段清晰可见吧，json具有同样优点，但是会有额外键名/序列化-反序列化的开销，在数据量偏少但是数据项偏多时这种额外开销显得到尤为繁重；protobuf解决了前者的问题，但是阅读性不好，内部是二进制格式，不方便我学习调试，且前置步骤很多，需要编写 .proto 协议文件定义数据结构，然后再靠工具生成对应类，一般用于跨语言开发(json也能跨语言开发)；xml的问题也是冗余字段，字符串形式贴近html格式，适合web服务，也有序列化开销。但是并不是说自定义二进制格式就没有缺点，而且特定情况会很严重，就是结构体内部对齐的问题，不同操作系统中对齐格式有可能不统一，导致解析出错，我当前的协议内部是主动4字节对齐的，暂时不会出现这种问题，如果出现对不齐的情况需要显式指定对齐策略(紧凑排列，n字节对齐)但是这样对不齐内存地址，回传出现两次io才能得到一个值的情况，所以说是有取舍的，当然最终迭代是要改为protobuf的，是现代通用标准(工具protoc)

### 追问：数据完整性做验证了吗？数据损坏了怎么办？

因为架构改为tls握手加bcrypt了嘛，那我就可以拿之前用的md5去做一个哈希摘要，然后服务端截断md5数组(放在末尾，直接根据定长截去)，对截断后的结构体进行哈希计算，再与哈希值进行比对，比对成功数据有效

### Q29:tcp收发会有什么问题？你是怎么解决的?

tcp字节流传输，会存在粘包问题，我采用的解决方式是先传包大小再传包数据，这样每次接收先固定接收4字节int大小包长然后申请空间，下发数据包。横向对比其他解决方式的话，还有短连接，固定包长度，特殊标志位等，短连接问题是反复创建销毁，开销太大，服务端如果几千个tcp连接同时销毁性能影响过大；固定包长度会有包碎片，而且超出限定大小的包还要有额外处理逻辑，得不偿失；特殊标志位由于包是二进制格式传输，有可能巧合出现于特殊标志位相同的字段从而错误截断

### Q30:断线重连你是怎么设计的?

## 断线重连

### 设计目标

- **软断线**（TCP 断开、60s 内）：保留 `USER_INFO` + token，客户端发 `RECONNECT_RQ` 恢复大厅/专区/房间/棋局，无需重新输密码。
- **硬断线**（距上次心跳超过 60s）：`delete USER_INFO`，重连失败，须重新登录。
- **身份识别**：入站靠包内 `userid` + `token`，不靠 fd；出站 `SendData` 用 `USER_INFO.m_sockfd`。

---

### 三层状态机

```
┌─────────────┐
│   在线      │  m_sockfd >= 0，心跳持续刷新
└──────┬──────┘
       │ TCP 断开 → OnFdClosed
       ▼
┌─────────────┐
│  软离线     │  m_sockfd = -1，USER_INFO / token / roomid 保留
│  (可重连)   │  仍在 m_roomList、room_cache
└──────┬──────┘
       │ 距上次心跳 > 60s → HandleDisconnect
       ▼
┌─────────────┐
│  硬离线     │  delete USER_INFO，须重新登录
└─────────────┘
```

#### m_sockfd 语义

| 值                           | 含义                |
| ---------------------------- | ------------------- |
| `>= 0`                       | 在线，合法 epoll fd |
| `-1` (`_DEF_OFFLINE_SOCKFD`) | 软离线，可重连      |
| （USER_INFO 已 delete）      | 硬离线              |

```cpp
#define _DEF_OFFLINE_SOCKFD  (-1)
#define IS_ONLINE_SOCKFD(fd) ((fd) >= 0)
```

### 网络层关闭（block_epoll_net.cpp）

#### 统一入口 `CloseFd(fd, notify)`

```cpp
void CloseFd(sock_fd fd, bool notify = true)
{
    if (!m_mapSockfdToEvent.find(fd, ev)) return;  // 已关过则直接返回

    if (notify && m_closeCb) m_closeCb(fd);   // → OnFdClosed

    ev->eventdel();
    close(fd);
    m_mapSockfdToEvent.erase(fd);
    delete ev;
}
```

#### 调用方与 notify

| 调用方                         | notify  | 原因                       |
| ------------------------------ | ------- | -------------------------- |
| `recv_task`（对端断 TCP）      | `true`  | 需 `OnFdClosed` 标软离线   |
| `HandleDisconnect`（心跳超时） | `false` | 已 `delete USER_INFO`      |
| `ReconnectRq`（换绑 fd）       | `false` | 业务自行更新 `m_sockfd`    |
| `LoginRq`（踢旧连接）          | `false` | 即将 delete 旧 `USER_INFO` |

#### recv_task 失败主路径

```
read/recv <= 0
  → CloseFd(fd, true)
  → OnFdClosed(fd)
  → 网络资源释放
```

#### 5.1 OnFdClosed（软离线）

```cpp

// 不清 m_heartbeatMap，不清 m_roomList
```

#### 5.2 HandleDisconnect（硬离线）

```cpp
erase m_mapfdtouserinfo;
erase m_fdToUserid[old_fd];
erase m_heartbeatMap;

if (IS_ONLINE_SOCKFD(old_fd)) CloseFd(old_fd, false);
// 注意：暂未从 m_roomList 移除（幽灵玩家，待下阶段）
```

#### 5.3 HeartBeatRq / CheckHeartBeat

- 收到心跳：`m_heartbeatMap[userid] = now`
- 每 10s 扫描：`now - last > 60` → `HandleDisconnect`

#### 5.4 LoginRq（重复登录）

替换旧会话时：

```cpp
erase m_fdToUserid[old_fd];
if (old_fd != clientfd) CloseFd(old_fd, false);
delete 旧 USER_INFO;
新建 USER_INFO，生成新 token，绑定 clientfd
```

#### 5.5 ReconnectRq

```
1. 查 userid → 无则 result=0
2. strcmp(token) → 失败 result=0
3. 绑新 fd：m_sockfd=clientfd，更新 m_fdToUserid，续心跳
4. zoneid=info->m_zoneid，roomid=info->m_roomid
   inGame = (room_cache.count(roomid) > 0)
5. 发 RECONNECT_RS

6. if roomid > 0：
   ├─ 重放 STRU_ROOM_MENBER（按房间列表顺序算 status）
   ├─ 终止标记 { userid = -1 }
   ├─ if inGame==0 且重连者是 host/player：
   │     向其他在线 host/player 广播 DEF_FIL_ROOM_NOTREADY { userid=重连者 }
   └─ if inGame==1：
         按 room_cache 顺序重放 STRU_FIL_PIECEDOWN { color=step%2, x, y }
```

#### 5.6 出站发送防护

房间广播 / 落子转发 / 离开通知等处：

```cpp
if (!IS_ONLINE_SOCKFD(info->m_sockfd)) continue;
SendData(info->m_sockfd, ...);
```

#### 5.7 JoinRoomRq / LeaveRoomRq

- 加入：`info->m_roomid = rq->roomid`
- 离开：`info->m_roomid = 0`

## 端到端时序

### 7.1 正常断线重连（房间内未开局）

```
客户端                         服务端
  │ TCP 断                      │
  │────────────────────────────>│ recv_task → OnFdClosed → m_sockfd=-1
  │ 停心跳                       │
  │ 弹窗点「是」                  │
  │ OpenNet + RECONNECT_RQ      │
  │────────────────────────────>│ ReconnectRq：校验 token，绑新 fd
  │<────────────────────────────│ RECONNECT_RS
  │<────────────────────────────│ ROOM_MEMBER × N
  │<────────────────────────────│ ROOM_MEMBER {userid=-1}
  │                             │ → 向对方发 NOTREADY(重连者)
  │ slot_reconnectRs 恢复房间    │
  │ 对方 delPlayerReady          │
```

### 7.2 对局中重连

```
RECONNECT_RS inGame=1
→ ROOM_MEMBER 重放
→ {userid=-1} → setGameStart()
→ PIECEDOWN × N 恢复棋盘
```

### 7.3 超时硬离线

```
断线后停心跳 → 60s 无心跳 → HandleDisconnect → delete USER_INFO
→ RECONNECT_RQ → result=0 → 客户端回登录
```

设计要点速记

1. **重连有意义的前提**：TCP 断只软离线（`m_sockfd=-1`），不 delete `USER_INFO`；超时才硬删。
2. **不要用 0 表示离线**：0 是 stdin；用 `-1`。
3. **`m_fdToUserid`**：仅 `OnFdClosed` 反查 userid；与 `USER_INFO.m_sockfd` 在线时重复维护。
4. **身份在包里**：`userid` + `token`；fd 复用不影响入站；出站靠 `IS_ONLINE_SOCKFD` 防误发。
5. **重连窗口**：断线后客户端停心跳，实际窗口 = 距**最后一次心跳** ≤ 60s。
6. **`CloseFd(notify)`**：网络清理统一走一处；业务是否感知由 `notify` 控制。

### Q31:详细说一下相对于经典观察者模式的区别?

### 1. 支持多对多绑定

1标准观察者一般是一对多；

信号槽支持：

- 一对多：一个信号绑定多个槽；
- 多对一：多个信号绑定同一个槽；

2. 原生跨线程支持（标准观察者无线程能力）

自动区分 Direct/Queued/Blocking 连接，跨线程封装事件队列投递，线程安全分发；普通观察者需要自己加锁、处理线程同步。

3. 参数传递能力

信号可以携带任意可拷贝数据传给槽，标准观察者回调大多无参或固定参数。

4. 自动生命周期管理

QObject 销毁自动清除所有相关连接，自动注销观察者，杜绝悬空指针崩溃；手写观察者模式极易忘记反注册导致野指针。

5. 四种连接调度策略

直连、队列、阻塞队列、唯一连接，灵活控制同步 / 异步执行；原生观察者只有同步调用。

6. 元对象系统驱动，动态绑定

运行时可动态 connect/disconnect，不需要编译期固定接口；传统观察者接口强耦合。

### Q39:极大极小搜索树是怎么用的?

整体描述:首先说为什么要使用它。五子棋是零和博弈，天然适合用极大极小搜索树这种博弈树进行计算。他是以电脑视角展开的，因此max层对应电脑获取所有子树中收益最大化的，min层对应玩家获取所有子树中对电脑收益最小化的，然后就是深度问题，太浅计算结果不够精准，太深时间复杂度上天(O(b^d))b为分支数量，d为递归深度，所以平衡取舍下我选择了4层，包括两次max计算两次min计算，但是这种情况下还是会卡顿，因此我搭配上了ab剪枝(懒的打准确符号了)，剪枝生效的情况是需要不同层级参与的而不是发生在同一层级下，我简单举个例子，比如最后的min层有很多个节点第一个节点通过对其子节点(叶子节点)的计算(min)得到一个值为a，对第二个节点进行计算时得到一个值为b，依次类推直至计算完毕(因为min层，只会对beta值进行更新)，因为他们的父节点是max层，计算就取最大值，此时会得到alpha值，再进入max层的第二个子节点(min层)，此时就有情况触发alpha-beta剪枝了，min层取第一个子节点的值时会更新beta值，因为此时从max层已经带下来一个alpha值了，此时alpha>beta就会触发剪枝，后续计算没有必要了。至于为什么是这样，我们可以用min和max替换alpha/beta，min的含义是父层至少能收获的收益，max含义是当前层能获得的最大收益，如果max<min了，那当前层再怎么计算，父层都不会选择此节点，因此计算都是没有必要可以省去。达到了降低复杂度的结果。对min层排序从小到大，对max层排序从大到小

极大极小搜索树是递归(dfs)，递归最关键的就是确定参数，分别是当前棋盘状态，数组存储的每一步落子，因为要根据他进行候选点的计算，当前深度，alpha值，beta值，当前层级(max/min)，以谁的视角计算。当然只有ab剪枝不够，还需要搭配其他优化才能让效果足够使用，我采用的方法就是对每层节点进行排序，max层从大到小排列，min层从小到大排列，这样更容易让alpha>beta，触发剪枝

然后就是计算流程

最上层因为要进行点的选择，所以需要从递归代码中摘出来，剩余层进行递归，效果是逻辑更清晰，坏处是重复代码。找到预选点，先判断双方下此点是否会直接输赢是则直接返回没必要继续计算，否则假设下此点，然后进行minmax评估(几乎重复逻辑)，得到此点分数，继续评估下一个点，然后找出最优点，然后另一个最关键的东西是评估，会根据当前棋盘所有的点分为4个方向展开算是否连珠，根据活/死-4/3判断分数，然后返回。

压测怎么做的?

分为长连接压力与并发处理压力，我从1200逐步增加压测数量，在50并发的前提下，长连接都可以稳定运行，主要瓶颈在登录的bcrypt校验和数据库单连接的串行化执行上，回复超过10s被判超时失败，最后大致稳定通过的是4800长连接与50左右并发数(因为每个已经进行的长连接还要发包刷新心跳，后面连接越多开销越大，机器已经明显卡顿，就停手了)

线程池参数？

帧同步服务器，瓶颈在I/O上，与核心数有关，一般是核心数*(1+i/o阻塞耗时和cpu计算耗时的比值)，我设置为初始10最大200，以及额外一个管理者线程，用来扩容缩容，扩容条件是忙线程超过80%，或者任务数量超过空闲线程数，缩容条件是忙碌线程不超过1/3且当前线程数不小于最少线程数，任务队列我设置为了50000

帧同步与状态同步的区别？

一个只负责转发，通过客户端执行相同逻辑得到相同结果；另一个所有计算都在服务端进行，每帧由服务器转发当前状态，数据更安全，但是服务器计算压力很大。

有考虑过其他的搜索树吗?

我了解过其他搜索树，比如蒙特卡洛斯树，核心思想是随机采样模拟，然后聚焦于高胜率点继续展开，然后对胜率进行统计判断局面，每个节点存总访问次数和累积胜利次数，胜率就是两者比值，他有四步循环，选择->扩展->模拟->反向回溯，循环成千上万次四步迭代后，根节点下**访问次数最多的子节点**就是最优落子，无序复杂评估函数，且动态非对称展开树，反复对好分支进行计算，还能通过神经网络改善纯随机模拟。但是对于五子棋这种轻型游戏劣势很大，首先五子棋的有效落子很少，分支很少，能探索到更深层；蒙特卡洛适合分支过多随机选中，深度也做不到很广，且五子棋有明确的评分标准，与MCTS不契合，五子棋的胜负链条短，就是几部之内就能决定胜负，随机模拟很难精准捕捉这种局面陷阱，成本也相对于极大极小搜索树更大
