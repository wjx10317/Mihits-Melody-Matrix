## 1. 视觉与渲染

### 1.1 列拓展视觉过渡效果

**状态**: 待确认
**潜在价值**: 高 — 列拓展是矩阵动态变化的核心视觉表现
**实现复杂度**: 中高

**背景**:
行拓展的视觉过渡已确定方案（Insert渐入 / Rotate旋转），但列拓展的过渡方式尚未确认。

**候选方案**:

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A. 侧方渐入 | 新列从右侧/左侧淡入插入 | 与行拓展的Insert方案一致 | 列数增加时视觉不够震撼 |
| B. 分裂扩展 | 现有列向两侧分开，新列从中间插入 | 视觉冲击力强 | 实现复杂，需要处理每列的独立动画 |
| C. 旋转（同行拓展） | 统一使用旋转过渡 | 风格统一 | 列变化用旋转可能不如行变化自然 |
| D. 缩放弹出 | 整个矩阵先缩小，新列弹出后恢复 | 简洁有力 | 可能影响正在进行的Hold音符视觉 |

**建议**: 先实现方案A（与Insert一致），后续可增加更多过渡类型供谱师选择。

## 2. 游戏玩法

### 2.1 Mod 系统

**状态**: 部分接口已定义，未完整实现
**潜在价值**: 高 — 增加可玩性和重玩价值
**实现复杂度**: 中

**已有基础**:
- `IJudgeStrategy` 接口 + `EasyJudgeStrategy` 实现
- `IHpMod` 接口 + `NoFailMod` 实现
- SongSelect 中有 Mod 弹窗 UI

**待实现**:

| Mod | 效果 | 实现难度 |
|-----|------|----------|
| Easy | 判定窗口×1.4（已有策略类） | 低 |
| NoFail | HP不会归零（已有模组类） | 低 |
| HardRock | 判定窗口×0.7，HP消耗×1.4 | 低 |
| DoubleTime | 播放速度×1.5，AR调整 | 中 |
| HalfTime | 播放速度×0.75，AR调整 | 中 |
| Hidden | 音符接近时渐隐 | 中 |
| Flashlight | 可见区域限制 | 高 |
| Mirror | 列左右翻转 | 低 |
| Random | 列随机重排 | 中 |

**关键问题**:
- DoubleTime/HalfTime 需要音频变速播放（miniaudio是否支持？）
- Hidden/Flashlight 需要shader支持
- Mod组合的交互规则（如 HD+HR 是否允许？）

### 2.4 歌曲结束判定优化

**状态**: 当前仅检查 `!m_audio.isPlaying() && m_audio.positionMs() > 0`

## 6. 性能与架构

### 6.1 NotePool / NoteFactory 集成或移除

**状态**: 已实现但从未使用
**潜在价值**: 低-中
**实现复杂度**: 低

**选项**:
- A. 移除：当前使用 vector<Note> 拷贝，对象池无必要
- B. 集成：JudgeQueue 使用 NotePool 管理音符生命周期
- C. 延迟决策：保留代码，待性能分析后决定

**建议**: 选项C，当前不影响功能。

---

### 6.2 EffectRenderer 性能优化

**状态**: 已实现但未集成
**潜在价值**: 中
**实现复杂度**: 中

**待优化**:

| 问题 | 当前实现 | 优化方案 |
|------|----------|----------|
| 粒子删除 | `erase(begin())` O(n) | 环形缓冲区或swap-and-pop |
| RNG | 固定种子 `mt19937(42)` | `random_device` 初始化 |
| 粒子上限 | 硬编码512 | 可配置 |
| 批量渲染 | 每帧重建vertex data | 增量更新 |

---

### 6.3 NoteRenderer 可见窗口裁剪

**状态**: 遍历全部音符
**潜在价值**: 中
**实现复杂度**: 低

**方案**: 利用 JudgeQueue 的 headIndex 和 approachTime 做范围裁剪，只遍历可见范围内的音符。

---

### 6.4 InputManager 正确集成

**状态**: 已实现但 PlayingState 直接使用 SDL_GetKeyboardState
**潜在价值**: 中
**实现复杂度**: 低

**方案**:
- 在 Kernel::pumpInputEvents() 中将游戏按键事件转发给 InputManager
- PlayingState 通过 InputManager 的回调接收按键事件
- 移除直接 SDL_GetKeyboardState 调用

**注意**: 边沿检测仍需在 PlayingState 中实现（InputManager 只负责事件分发）。

---

### 6.5 事件总线利用

**状态**: EventManager 已实现但 PlayingState 未使用
**潜在价值**: 低-中
**实现复杂度**: 低

**说明**: 当前判定事件通过直接函数调用处理。如果未来需要多个系统响应判定事件（HUD更新、特效、音效、统计等），应切换到事件总线模式。
