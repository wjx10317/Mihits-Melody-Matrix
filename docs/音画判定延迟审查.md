# Melody Matrix 音画判定延迟审查

> 来源：Cursor Canvas `music-game-latency-review.canvas.tsx` 同步副本（防上下文丢失）。  
> 权重：**源码事实优先**；旧「PCM frame anchor → 2–3ms」文档结论已降级（见 [`音频时钟-cursor台阶优化.md`](./音频时钟-cursor台阶优化.md)）。  
> 环境假设：Windows / SDL2 2.30.8 / miniaudio 0.11.21  
> 同步日期：2026-07-17

---

## 结论（先读）

**不能对任意设备承诺端到端 ±2–3ms。**

当前代码把 **数据源读取 cursor** 当作耳朵听到的时间，又用 **固定视觉提前量** 猜测显示延迟。应用层可以把「内部判定相对设备播放时钟」做到约 2–3ms；但蓝牙、未知驱动/扬声器 DSP、显示器处理、键盘硬件扫描无法被纯软件完整观测，零用户操作下无法普适保证。

| 指标 | 现状 |
|------|------|
| miniaudio 默认 period | ~10ms（仅 hint，实际由后端协商） |
| Clock / SDL 事件时间粒度 | 1ms |
| 60Hz 默认视觉 lead | ~24ms（16ms + 半帧） |
| 自动化时序测试 | 0 |

**最容易误判的主张：**「改成 PCM frame 就能降到 2–3ms」**不成立**。`positionMs()` 已经先读 PCM frame；改为保存 frame 只移除约 ±0.5ms 的 ms 四舍五入。cursor 仍按 callback 块推进，且位于设备缓冲之前。真正需要的是 **device presentation position + 与高精度主机钟关联的采样时刻**。

---

## 当前实际链路

```
谱面 note(ms)     → 目标歌曲时间
ma_sound cursor   → 数据源已读取帧（不是听感）
Clock anchor      → 主机/SDL ms 外推
按键事件          → SDL ms timestamp（方案 B 起改为 HostClock 处理时刻）
判定              → press − note
渲染              → +16ms + 半帧 visual lead
```

## 问题清单与修改方向

| 优先级 | 域 | 问题 | 机制/证据 | 影响 | 修改方向 |
|--------|----|------|-----------|------|----------|
| P0 | 音频 | BGM cursor 不是耳朵听到的位置 | `ma_sound` cursor 是 data source 已读取帧；后面还有混音、WASAPI、驱动、DMA/DAC | 固定偏差 + 抖动，数 ms～数十/数百 ms | Windows 读 `IAudioClock`/`IAudioClock2` + 主机钟；按设备 epoch 映射到歌曲帧 |
| P0 | 音频 | 默认设备配置完全交给 miniaudio | `ma_engine_init(nullptr)`，默认 period 10ms，实际值由后端协商 | 台阶约一个 period；实际缓冲未知 | 自建 `ma_device`/engine；显式 WASAPI、原生格式、2–3ms 目标 period、2–3 periods，并记录协商值 |
| P0 | 音频 | Clock 在「观察 cursor 的主线程时刻」打 anchor | cursor 所属帧没有关联的高精度时间戳；跳变后以当前 tick 硬锚定 | 0～一个 callback 周期的相位误差 | 采集 `(deviceFrame, hostTime)`；线性/PLL 外推，拒绝异常样本，seek 时新建 epoch |
| P0 | 输入 | SDL 2.30.8 timestamp 不是硬件中断时间 | `Uint32` 毫秒 tick；Windows 事件在 SDL pump/入队链路产生 | VSync 下可附加 0～1 帧，60Hz 最坏约 16.7ms | Raw Input/消息原始时间映射主机钟，或验证 SDL3 ns；输入与 Present 解耦 |
| P0 | 显示 | 固定 visual lead 不是实际呈现延迟 | 当前 lead = 16ms + 上一帧时长/2 | 60Hz ~24ms、144Hz ~19ms，与真实 DWM/扫描无关 | 预测 present 主机时刻；DXGI flip + latency waitable + frame statistics |
| P0 | 显示 | SwapWindow/VSync 阻塞输入与时钟采样 | `SDL_GL_SetSwapInterval(1)`，主线程 Swap 后才继续泵事件 | 按刷新率成批处理，抖动可达一帧 | 渲染前后独立泵输入；最终独立输入线程/低延迟 swapchain |
| P0 | 启动 | Clock reset 早于解析、加载和实际开声 | reset 在谱面解析前；`ma_sound_start` 异步，cursor=0 时 anchor 不刷新但 Clock 已外推 | 首段时间跳变 | 预缓冲后按设备钟安排开播；首个有效 device position 后才开放判定 |
| P1 | 暂停/Seek | pause/resume/seek 没有完成确认 | stop/start/seek 与音频线程异步；Clock 立即重锚 | 恢复/跳转瞬间阶跃 | 每次操作建 timeline epoch；设备钟确认后再原子切换 |
| P1 | 时钟 | 全链路整数毫秒 | frame→ms；SDL ticks/事件也为 ms | 约 ±0.5～1ms，叠加逼近预算 | 内部统一 ns/主机钟 + PCM frame；仅 UI/谱面边界转 ms |
| P1 | 时钟 | 无音频设备时钟漂移模型 | 外推默认与声卡晶振同速，靠硬重锚纠偏 | 长曲漂移 + 重锚抖动 | 在线估计 rate/offset；软跟踪小漂移，underrun 时硬重建 |
| P1 | 显示 | 没有真实呈现反馈 | OpenGL/SDL 只知道提交与 Swap 返回 | DWM/驱动排队可再加 1–2 帧 | 优先 D3D11/DXGI flip；保留 GL 则用 DWM timing 估计并标明降级 |
| P1 | 显示 | frameTime/2 可在卡顿后放大 visual lead | frameTime clamp 250ms → 补偿可瞬时 +125ms | 掉帧后一帧严重超前 | 补偿只来自预测呈现 deadline；稳健滤波 + 限幅 |
| P1 | 运行时 | 热路径 DEBUG 文本仍被构造 | 日志宏无调用前级别判断 | 分配/I/O 造成尾延迟 | 先 `isEnabled`；热路径无分配 ring-buffer telemetry |
| P1 | 音效 | 击打音也经过同一输出缓冲 | 按键后 `ma_sound_start`，只能在后续 callback 后听到 | 有线数～数十 ms；蓝牙更高 | 小 WAV 常驻、低延迟设备、callback 内 sample-accurate 调度 |
| P1 | 设备 | 未处理设备切换、重路由、采样率变化和 underrun | 默认 engine 隐藏后端状态 | 突然固定偏移、漂移或跳变 | 订阅 device notification；按 endpoint 重建模型；underrun 暂停评分 |
| P2 | 谱面 | 谱面与音频文件本身可能有偏移 | 解析器直接采用 note 毫秒 | 固定偏差随谱面变化 | 导入阶段规范化 codec delay；chart offset；onset 质检 |
| P2 | 配置 | 三套 offset 语义分裂 | `timing_offset` 生效；`visual_lead` 手动；Clock user offset / `gameplay.offset_ms` 基本未接线 | 双重或遗漏补偿 | 统一 LatencyModel，分字段，禁止一个总 offset 到处叠 |
| P2 | 测试 | 没有时钟/判定自动测试 | CMake 有 tests 开关，仓库无有效时序测试 | 2–3ms 目标不可验收 | 虚拟设备 + 确定性主机钟；seek/pause/device-change；硬件回环基准 |

## 建议的统一时间模型（LatencyModel）

不要再用一个总 offset 到处叠加：

| 字段/概念 | 含义 |
|-----------|------|
| `audioFrameAtHost(t)` | 设备正在呈现的歌曲帧（音频权威源） |
| `inputHostTime` | 输入实际发生时间 → 换算歌曲帧后判定 |
| `predictedPresentHost` | 预计本帧进入扫描显示的时间 → 渲染 note |
| `chartOffsetFrames` | 只描述谱面/文件固有偏移 |
| `unobservableOutputTail` | 蓝牙/外部 DSP 等，仅测量或降级，不伪造精度 |

主机钟抽象为 `time::HostClock`（Win = QPC）；听感位置为 `time::AudioPlayhead`（后接 WASAPI）。详见时序总计划文档。

## 实施顺序

1. **可观测性** — 先记录 source cursor、device clock、主机钟、padding、period、输入时间、present 统计与 underrun；禁止继续靠手感改常量。
2. **音频权威时钟** — Windows 落地 WASAPI `IAudioClock2`/`IAudioClock` + HostClock。歌曲时间由「当前正在播放的设备帧」导出，不再由 data-source read cursor 导出。
3. **输入同域** — 键盘事件在 HostClock 域时间戳化，再通过同一音频模型换算为歌曲帧；判定内部保持帧/ns 精度。
4. **呈现预测** — 以预计扫描显示时刻计算渲染时间；动态替代 16ms + 半帧；控制 swapchain 排队深度。
5. **自动配置** — 按 audio endpoint、采样率、显示器/刷新率、窗口模式、键盘设备保存 profile；设备变化自动失效并重建。
6. **验收与降级** — 内部残差目标 P95≤2–3ms；无法观测的蓝牙/显示器 DSP 标记为不可保证。

**与仓库里程碑对齐（`test` 分支）：** 文档 → HostClock 方案 B → Raw Input → AudioPlayhead/WASAPI → 自动校准。

## 可达性分层

| 层级 | 内容 |
|------|------|
| **可达到** | 有线 WASAPI、稳定设备、IAudioClock + HostClock、低排队显示路径时，内部音频↔输入映射残差可争取 P95 2–3ms |
| **只能估计** | 普通 OpenGL+DWM 的 photon time、键盘扫描/去抖、扬声器声学传播 — 可 profile/统计，非精确观测 |
| **不能保证** | 蓝牙耳机、USB 无线 dongle 缓冲、电视/显示器图像处理、第三方虚拟声卡 — 无外部回环时不能承诺 ±2–3ms |

---

## 验收标准

**软件内：** 虚拟设备注入已知 device-frame/主机时刻，覆盖启动、暂停、恢复、seek、underrun、设备切换和 30 分钟漂移；误差看 P50/P95/P99，不看单次平均值。

**端到端：** line-out→line-in 回环测音频，光电二极管测显示，键盘执行器/高采样逻辑分析仪测输入。按 endpoint、刷新率、窗口模式分别报告。

**发布门槛：** 有线参考设备 P95≤3ms；未知或无线链路明确降级，不通过隐藏常量把一台机器的结果冒充通用保证。
